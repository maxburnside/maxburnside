I'm Going To Build an App

November 8, 2023

As 2023 winds down, there is discussion of how technology such as AI is changing our world. That's a valid discussion point for a different post, as it certainly is changing how we interact with media, computers, and information. What I'm more surprised by, rather, is how a decades old technology has managed to stay relevant all these years later. Short Message/Messaging Service or SMS (commonly referred to as "texting") has been around since the early 90's. Over 30 years since it's inception, it's still consistently utilized by nearly every large company in the industrialized world.

But why?

As is usually the case, there are a few reasons:
  • Cost
  • Simplicity
  • Reach
  • Effectiveness

With such a ubiquitous technology, I figured there must be an industry leading tool to check how text messages will be formatted. That was an incorrect assumption.

While there are a number of services that allow you to send text messages to customers, none of them provide an exact representation of the formatting that users will see when the message gets to them. The previews provided are abstracted phone displays. When it comes to paying for every message (however small the financial cost) you're sending, you should know more precisely how they're going to look. With different screen sizes, font sizes, appearance settings and operating systems, how your message will look when the recipient receives the message can vary greatly.

Discovering this void, I figured, "Why don't I just make this application?"

I'm first and foremost a designer. Over the years, I’ve learn to code well enough to build out my design ideas. I started, as many do, with coding static websites (like the one that's hosting this blog post you're reading). Working at a variety of startups, wherein there was either a small development team, or just one developer, I found an increasing value in demonstrating how the UI should be implemented. At certain companies, I was “just a designer”, while at others I was actually coding the UI myself.

But beyond the UI, I was interested in creating the entire application. That’s the journey I’m still on, and will be for some time.

I'm an autodidact. I teach myself. It often takes longer. It's a messy path to knowledge. But it also ensures I better internalize lessons. As with any skill, the more you perform it the better you get (typically).


Technology Specifics


The first iteration of the app will web-based. There are a few reasons for this:

  • Ease of publishing updates (no app store approval delays)
  • Lowest barrier to entry for users. I’m planning to use X, Reddit, Threads and Instagram to share the site URL, and I want users to be able to jump right in.
  • The planned experience works best when using a larger screen size, which are more readily available in web formats. While not specifically relegated to desktop users, designing a web application will make it possible for users on both Mac and Windows to preview multiple phone previews simultaneously.
  • Ease of building quickly. While I have some experience in Objective-C and Swift, I have far more experience with HTML, CSS, and JS frameworks (the first iteration of the app will be built with React), which will make building the first version of the application much easier.
